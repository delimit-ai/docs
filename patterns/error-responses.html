OpenAPI Error Response Pattern

Use RFC 7807 Problem Details for consistent, machine-readable error responses. Define them once in OpenAPI, reuse everywhere, and let Delimit protect the contract.

RFC 7807 Problem Details Schema

RFC 7807 (updated by RFC 9457) defines a standard JSON format for HTTP API error responses. It provides five core fields that every error should include.

# OpenAPI 3.0 — RFC 7807 Problem Details
components:
  schemas:
    ProblemDetails:
      type: object
      required:
        - type
        - title
        - status
      properties:
        type:
          type: string
          format: uri
          description: URI reference identifying the problem type
          example: "https://api.example.com/problems/insufficient-funds"
        title:
          type: string
          description: Short human-readable summary
          example: "Insufficient funds"
        status:
          type: integer
          description: HTTP status code
          example: 422
        detail:
          type: string
          description: Human-readable explanation specific to this occurrence
          example: "Account balance is $5.00 but transfer requires $10.00"
        instance:
          type: string
          format: uri
          description: URI reference identifying the specific occurrence
          example: "/transfers/a]b3c4d5/errors/1"

Extending Problem Details

RFC 7807 is designed to be extended. Add domain-specific fields alongside the standard ones.

# Extended error with validation details
components:
  schemas:
    ValidationError:
      allOf:
        - $ref: '#/components/schemas/ProblemDetails'
        - type: object
          properties:
            errors:
              type: array
              items:
                type: object
                required:
                  - field
                  - message
                properties:
                  field:
                    type: string
                    example: "email"
                  message:
                    type: string
                    example: "must be a valid email address"
                  code:
                    type: string
                    example: "invalid_format"

Using Error Schemas in Responses

paths:
  /transfers:
    post:
      summary: Create a transfer
      requestBody:
        required: true
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/TransferRequest'
      responses:
        '201':
          description: Transfer created
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/Transfer'
        '400':
          description: Invalid request
          content:
            application/problem+json:
              schema:
                $ref: '#/components/schemas/ValidationError'
        '422':
          description: Business rule violation
          content:
            application/problem+json:
              schema:
                $ref: '#/components/schemas/ProblemDetails'
        '500':
          description: Internal server error
          content:
            application/problem+json:
              schema:
                $ref: '#/components/schemas/ProblemDetails'

Common Mistakes

Mistake Problem Fix
Different error formats per endpoint Clients must handle multiple shapes Use a shared ProblemDetails schema everywhere
Using application/json for errors Clients cannot distinguish errors by content type Use application/problem+json
String-only error messages Not machine-readable, hard to localize Use structured type URIs and code fields
Missing status in body Proxies may change HTTP status; body should be self-describing Always include status in the response body

Breaking Changes Delimit Catches

Error responses are part of your API contract. Consumers depend on error schemas for retry logic, user-facing messages, and monitoring. Delimit detects:

Change Detection Severity
Remove detail field from error schema field-removed Breaking
Remove 422 response entirely response-removed Breaking
Change error type from string to integer type-changed Breaking
Add required trace_id field to error schema required-field-added Breaking
Add new 429 response response-added Non-breaking
Add optional help_url to error schema optional-field-added Non-breaking
