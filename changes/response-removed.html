Change Types » response_removed

response_removed

Breaking Severity: high

A success response (2xx) has been removed from an operation. Clients expecting this response status code will not handle the new response correctly.

Why Is This Breaking?

Consumers build their error handling and response parsing around documented status codes. Removing a 2xx response means clients may not properly handle the new response, leading to incorrect application behavior.

Example

Before

responses:
  "200":
    description: Success
    content:
      application/json:
        schema:
          $ref: "#/components/schemas/User"
  "204":
    description: No content

After

responses:
  "200":
    description: Success
    content:
      application/json:
        schema:
          $ref: "#/components/schemas/User"
  # 204 response removed

How Delimit Detects It

Delimit compares response status codes for each operation. Only 2xx response removals are flagged as breaking since error response changes are generally non-breaking.

Migration Guide

1. Document the new response behavior clearly. 2. If consolidating responses, ensure the remaining response covers all use cases. 3. Communicate the change in advance so consumers can update their response handling. 4. Consider versioning the API if the response semantics change significantly.
