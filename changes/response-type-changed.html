Change Types » response_type_changed

response_type_changed

Breaking Severity: high

The type of a response schema has changed (e.g., from object to array, or string to integer). All consumers parsing this response will break.

Why Is This Breaking?

Response type changes are the most impactful breaking changes for consumers. Every client that deserializes the response based on the old type will fail. This causes runtime errors in production.

Example

Before

responses:
  "200":
    content:
      application/json:
        schema:
          type: object
          properties:
            users:
              type: array

After

responses:
  "200":
    content:
      application/json:
        schema:
          type: array
          items:
            $ref: "#/components/schemas/User"

How Delimit Detects It

Delimit performs deep schema comparison on response bodies. When a type mismatch is found within a response context (identified by status code in the path), it emits response_type_changed in addition to the general type_changed.

Migration Guide

1. Never change response types in-place. Create a new endpoint version. 2. If wrapping/unwrapping a response, keep the old format and add a new field. 3. Use content negotiation (Accept header) to serve both formats. 4. Provide SDK migration guides for each supported language.

npx delimit-cli lint --old old-spec.yaml --new new-spec.yaml
