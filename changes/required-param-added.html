Change Types » required_param_added

required_param_added

Breaking Severity: high

A new required parameter has been added to an existing operation. Existing clients that do not send this parameter will receive validation errors.

Why Is This Breaking?

Existing consumers do not know about the new parameter and will not include it in their requests. The server will reject these requests with a 400 Bad Request error, breaking all existing integrations.

Example

Before

paths:
  /users:
    get:
      parameters:
        - name: limit
          in: query
          required: false
          schema:
            type: integer

After

paths:
  /users:
    get:
      parameters:
        - name: limit
          in: query
          required: false
          schema:
            type: integer
        - name: organization_id
          in: query
          required: true
          schema:
            type: string

How Delimit Detects It

Delimit compares the parameters arrays for each operation. Any new parameter with required: true that was not present in the old spec is flagged.

Migration Guide

1. Make the new parameter optional with a sensible default value. 2. If the parameter must be required, version the endpoint (e.g., /v2/users). 3. Communicate the change to consumers before enforcing it. 4. Consider using a header or context-based default for backward compatibility.

npx delimit-cli lint --old old-spec.yaml --new new-spec.yaml
