Change Types » format_changed

format_changed

Breaking Severity: high

The format of a field has changed (e.g., from date-time to date, or int32 to int64). Consumers parsing this value with format-specific logic will break.

Why Is This Breaking?

Many consumers use format information to choose parsing strategies. Changing from date-time to date or from uuid to plain string can break validation, storage, and display logic.

Example

Before

properties:
  created_at:
    type: string
    format: date-time

After

properties:
  created_at:
    type: string
    format: date

How Delimit Detects It

Delimit compares the format property on schema fields. A change in format value between old and new specs is flagged as breaking.

Migration Guide

1. Add a new field with the new format rather than changing the existing one. 2. If the format change is widening (e.g., date to date-time), it may be safe but should still be communicated. 3. Provide both formats during a transition period. 4. Update all SDK code generators after the change.

Detect This Change

npx delimit-cli lint --old old-spec.yaml --new new-spec.yaml

Or add the GitHub Action to catch this automatically on every pull request.
